# Plano — Card 6.4: Worker de processamento (Fase 6, Fila Assíncrona)

**Data:** 2026-06-21 · **Autor:** @planner v2.1 · **Tamanho:** G (3–5 dias, ~600–1000 LoC) · **Tipo:** feature (SEGURANÇA)

## Demanda
Lado consumidor do LRO: processo separado (`src/worker.ts`) instancia o `Worker` do BullMQ, lê `{jobId}`, recarrega o Job do Postgres (fonte da verdade), processa idempotente e seguro (download inputs → parse → merge → upload output), gerencia status e purga inputs. Hoje só existe a `Queue` (produtor 6.2/6.3), nenhum `Worker`.

## Gap descoberto (não estava no briefing)
O `StorageAdapter` **não tem método para gravar o OUTPUT** — só `uploadJobInput`. Sem `uploadJobOutput`, o worker não fecha. Virou a Fase 1.

## Reconciliação de contradições (checklist do card precede decisões fechadas)
- **C-1 (Usage):** worker **NÃO** incrementa usage (quota já reservada no enqueue — A-QUOTA/D-3; incrementar = cobrança dobrada). FAILED no worker **NÃO** estorna (A-REFUND). Checklist do card está errada → corrigir aceite. Worker não escreve em `usage`.
- **C-2 (path):** ignorar `jobs/{tokenId}/{jobId}/output/result.xlsx`; usar `buildJobOutputPath({userId,jobId,ext,now:job.createdAt})` → `{userId}/{date}/{jobKey}/output.{ext}`.
- **C-3 (R2):** é Supabase Storage (legado do texto).

## Requisitos vinculantes (B-6.4.x)
- **B-6.4.1** (sweeper PENDING órfão): recomendado implementar no **6.7**; 6.4 entrega só o claim idempotente que torna re-enqueue seguro.
- **B-6.4.2** (late-enqueue): claim atômico `UPDATE ... WHERE status='PENDING'` → 0 linhas = no-op idempotente.
- **B-6.4.3**: paths só via key-builder ancorado em `Job.createdAt`.
- **B-6.4.4**: ext de `inputFiles.files[].ext` (metadata), nunca do fileName; parser recebe fileName sintético (`input-00.${ext}`) anti-spoofing.
- **B-6.4.5**: parse em `worker_thread` + `AbortSignal.timeout` + `terminate()` (6.10 folddado — ver D-2).

## Decisões de design
- **D-1 graceful shutdown:** `src/worker.ts` espelha `server.ts` — SIGTERM → `worker.close()` (drena job ativo) → `closeQueueConnection()` → `prisma.$disconnect()`. Conexão TCP SEPARADA via `createQueueConnection()`. Idempotência como rede de segurança.
- **D-2 (6.10 folddado):** `AbortSignal.timeout`+`Promise.race` NÃO interrompe `XLSX.read` síncrono no event loop (CPU-bound). Só `worker_thread.terminate()` cancela de verdade. Timeout cosmético = @security reprova. → worker_thread É parte do 6.4.
- **D-3 ordem (nada atômico Storage+DB):** claim atômico → ler inputFiles+createdAt → download (timeout) → parse(thread+timeout)+validateColumns+validateRowLimits → merge+generate → uploadJobOutput → UPDATE COMPLETED(output_format/size/completed_at) → purge best-effort (inputs_purged_at só se todos removidos, espelha M-03).
- **D-4 retry:** classificar — parse/validação/row-limit/zip-bomb/magic-bytes = **permanente** (FAILED+purge+`UnrecoverableError`, sem retry); Storage/DB/rede = **transiente** (throw normal, BullMQ retenta backoff; FAILED+purge só no `attemptsMade===attempts`).
- **D-5 path/fileName:** path via key-builder ancorado em createdAt; ext de metadata; fileName sintético no parser.
- **D-6 pool Postgres worker:** `connection_limit` pequeno por env do processo (confirmar Fly process group na Fase 7) OU `WORKER_DATABASE_URL`.
- **D-7 upload output idempotente:** `uploadJobOutput` com `upsert:true` (path determinístico, sobrescrever a própria saída em retry é seguro).

## Riscos (pre-mortem top 5)
1. Worker travado por ReDoS (concurrency=1 → fila para) → D-2 worker_thread.
2. OOM Fly 256MB com XLSX grande → concurrency=1 + cap linhas + terminate + teste de carga 30MB.
3. Retry duplica cobrança/output/purge → claim idempotente + upsert + classificação.
4. Job PENDING órfão (quota perdida) → sweeper no 6.7 + claim idempotente.
5. errorMessage vaza path/stack → catálogo genérico PT-BR + Sentry só interno.
