### 1.5 Detalhes das decisões fechadas

#### D#1 — DELETE confirmation
- Schema Zod: `z.object({ confirmation: z.literal("CONFIRM_DELETE_ALL") })` com `errorMap` retornando hint do payload exato.
- Padrão GitHub repo delete + AWS S3 wipe + Stripe delete account.
- **Residuais obrigatórios** (todos viram tasks abaixo):
  - **Idempotency-Key MANDATORY** (rejeitar request sem header) — middleware do Card #74 já existe, só `enforce: true` no decorator.
  - **Rate limit dedicado**: 1 req/5min por user + cap global anti denial-of-wallet (Supabase delete API é paga).
  - **Advisory lock Postgres** (`pg_advisory_xact_lock(hash(user_id))`) durante operação — evita race com upload concorrente criando órfão.
  - **`audit_log_legal` entry obrigatória** com count de objetos deletados, IP, UA, fingerprint.
  - **R-9 elevado pra ALTO** (signed URL viva 5min após DELETE — vazamento pós-delete).

#### D#2 — `PRO_RETENTION_DAYS`
- Reusar env existente. Default `30` no `.env.example`.
- `env.ts` superRefine: `z.coerce.number().int().min(1).max(365)`.
  - `min(1)`: abaixo quebra purge-on-disable (expira no passado).
  - `max(365)`: acima vira risco LGPD (retenção excessiva).
- Uso em service: `import { env } from '../../config/env'; const expiresAt = addDays(now, env.PRO_RETENTION_DAYS);`
- **Nunca** usar `process.env.PRO_RETENTION_DAYS` direto (Zod boot-validated SSOT).
- Log estruturado em `/disable`: `{ event: 'history.disable.scheduled', retentionDays: env.PRO_RETENTION_DAYS, expiresAt }` — auditoria forense pra DPO.
- Nota: @dba sinalizou possível discordância semântica com @planner ("retenção operacional" = "retenção pós-disable"?) — flag baixa, não escalada. Se produto quiser desacoplar no futuro, env nova `DISABLE_GRACE_DAYS` + waiver pra divergência.

#### D#3 — Admin auth (achado crítico do @security)
- **`Role` enum atual no schema é `FREE | PRO` (billing).** Adicionar `ADMIN` ao mesmo enum colaria authz com billing — refactor cross-cutting que infla #145 G além do limite.
- **B é o único caminho viável** com `WV-2026-006`.
- **9 mitigations obrigatórias** (não-negociáveis):

| # | Mitigation | Local | Tamanho |
|---|---|---|---|
| 1 | Zod no boot: `ADMIN_USER_IDS` parsed como `array<UUID>`, fail-fast em prod | `src/config/env.ts` superRefine | ~5 LoC |
| 2 | `min(1).max(5)` admins | `src/config/env.ts` superRefine | ~2 LoC |
| 3 | Verificar existência do user no BD em CADA request (cache 30s OK) | admin middleware | ~5 LoC |
| 4 | `session.revokedAt IS NULL` no admin check (reusa Card 2.x) | admin middleware | ~3 LoC |
| 5 | `audit_log_legal` entry **ANTES** da ação (padrão Card #150 D-1, AWAIT) | admin handler | ~5 LoC |
| 6 | Rate limit 5 req/min por admin + 20 req/min global | rate-limit config | ~3 LoC |
| 7 | `crypto.timingSafeEqual` na comparação de userId | admin middleware | ~3 LoC |
| 8 | Step-up reauth via `X-Admin-Confirm` header (hash timestamp+refresh token) | admin middleware | ~10 LoC |
| 9 | Card descoberta `roles-admin-enum` criado **antes do merge** | Trello Backlog | externa |

**Threat model resumido:**
- B é estritamente mais seguro que A no estado atual contra comprometimento de BD (vetor mais provável que comprometimento de Fly secrets).
- Comprometimento env = game over independente (Fly secrets é raio de RCE total já).
- Rotação atômica via `fly secrets set ADMIN_USER_IDS=... && fly deploy` (sem migration, sem invalidação de sessão por tokenVersion — gap pré-existente).
- A ganharia se houvesse audit trigger no UPDATE de `users.role` + invalidação por tokenVersion já implementados — neither exists today.

**Severity sem mitigations:** CRÍTICO. **Com mitigations 1-9:** MÉDIO (débito calibrado).

#### D#4 — Listagem opt-out
- `403 FORBIDDEN` com envelope `{ error: { code: "FEATURE_DISABLED", message, details: { feature: "history" } } }`.
- Mensagem PT-BR: `"Histórico desabilitado nas suas preferências. Dados anteriores serão purgados conforme política de retenção."`
- **Invariante cross-card:** `GET /history/:id` segue mesma regra (não 404 — vazaria existência por timing/diff; não 200 — contradiz opt-out). Documentar como invariante do módulo.
- **Adicionar** `FEATURE_DISABLED` ao `app-error.ts` ErrorCodes enum + `categories.json`.
- 200 com array vazio = falsa declaração ao titular do dado (LGPD: dever de informar estado real).

## 6. Decisões & trade-offs

### D-A — Path do scheduler: `src/scheduler/` (top-level)
Scheduler é cross-cutting infra (não "lib reusável"), tem rotas próprias (admin), facilita descoberta. Confidence: high.

### D-B — Soft delete: apenas `deletedAt nullable` (sem enum status)
Estado derivado: NULL=ACTIVE, NOT NULL=PURGE_PENDING. Hard-deletado pelo cron. Menos complexidade. Confidence: high.

### D-C — `expiresAt` calculado em service (não trigger DB)
Permite tunar via env sem deploy schema. Usa `env.PRO_RETENTION_DAYS` validado no boot. Confidence: high.

### D-D — Admin auth: JWT + `ADMIN_USER_IDS` env com `WV-2026-006` + 9 mitigations
**Atualizado 2026-05-02:** opção A original (adicionar ADMIN ao enum `Role`) é REJEITADA — `Role` no schema é billing (FREE/PRO). Refactor pra enum `UserRole` separado seria cross-cutting fora do escopo de #145. B com 9 mitigations é estritamente mais seguro que A no estado atual contra comprometimento de BD. Waiver kill criteria: enum `UserRole` separado + middleware unificado. Confidence: high (com mitigations).

### D-E — 1 lock POR JOB (não global)
Jobs independentes; bloquear quota-alert pq purge está rodando degrada UX sem ganho. Confidence: high.

### D-F — Confirmação destrutiva: string literal
`{ "confirmation": "CONFIRM_DELETE_ALL" }` (não boolean). Defesa contra erro silencioso de cliente buggado/curl/replay de payload genérico. Padrão GitHub/AWS/Stripe. Confidence: high.

### D-G — Listagem em opt-out: 403, não 200 vazio
`403 FEATURE_DISABLED`. 200 vazio mente sobre estado real (rows existem até purga). Invariante aplicada também em `GET /history/:id`. Confidence: high.
