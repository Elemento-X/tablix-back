# Card 6.7 — Cron de cleanup async + Sweeper #197 (Fase 6)

**Data:** 2026-06-22 · **Size:** G · **Tipo:** feature (cron/infra) + dívida robustez (#197)
**Plano:** @planner v2.1 · **Decisões fechadas com o dono** (sessão 2026-06-22)

## Escopo combinado (6.7 absorve #197)
Três classes de "estado preso" do caminho async, todas centradas na tabela `jobs` (SSOT):
1. **#197 — PENDING órfão da fila:** job criado PENDING cuja sequência `create → upload → enqueue` quebrou após reservar quota mas antes do BullMQ aceitar. Quota presa, nenhum worker processa. → re-enqueue OU FAILED+refund.
2. **6.7b — PROCESSING travado:** worker morto mid-flight + BullMQ esgotou retries → DB preso em PROCESSING. → FAILED (status-guard), **sem** refund.
3. **6.7a — cleanup de Storage (DB-driven):** purgar inputs de terminais com `inputs_purged_at IS NULL` (B-6.7.1) + outputs de jobs `expires_at < now()`.

## Decisões travadas com o dono
- **D-1 (orphan storage):** **DB-driven** (tabela jobs é SSOT; ninguém hard-deleta job; todo upload é pós-create). Varredura bucket-wide = card de backlog (safety-net futura).
- **D-2b (refund de PROCESSING travado):** **NÃO estornar** (alinha A-REFUND + FAILED de última tentativa do worker). Sweeper #197 (PENDING, nunca processado) **continua estornando**.
- **D-2a (limiar PROCESSING):** env `ASYNC_STUCK_PROCESSING_MINUTES` default 60, **gateado primariamente pelo cross-check da fila** (só FAILED se ausente/terminal na fila).
- **D-3 (purga):** **enumerar inputs individualmente** via `inputFiles` + `removeByPath` (NÃO delete-by-prefix; NÃO relaxar STORAGE_JOB_PATH_REGEX).
- **D-4 (sweeper):** re-enqueue se (dentro do TTL **E** fila ok **E** inputs presentes); senão FAILED+refund. Refund **idempotente por transição de status** (`updateMany WHERE status='PENDING'` → count===1) + **no período do `createdAt`** (`decrementUsageForPeriod`).
- **D-5 (cadência):** **2 crons** — `async-job-sweeper` (#197 + 6.7b) a cada **5min** (`*/5 * * * *`); `async-storage-cleanup` (6.7a) **diário 08:00 BRT** (`0 11 * * *` UTC). Kill-switch dedicado `CRON_JOBS_CLEANUP_ENABLED` + gate `ASYNC_PROCESSING_ENABLED` (NÃO acoplar a HISTORY).
- **D-6 (índices):** 3 índices parciais SQL-managed CONCURRENTLY (shape final @dba): `pending(created_at)`, `processing(started_at)`, `inputs_unpurged(completed_at)`. Outputs expirados reusam `idx_jobs_expires_at`.
- **A-4 (tombstone output):** o 6.6 seta `downloadedAt` + remove output, mas NÃO anula `outputFileUrl`. Predicado de purga de output expirado: `expires_at < now() AND downloaded_at IS NULL AND output_file_url IS NOT NULL`; pós-purga `SET output_file_url = NULL` (tombstone, sem coluna nova).

## Bindings vinculantes

## Riscos-chave (mitigados)
- R-1 force-fail de job legítimo → limiar alto + cross-check fila obrigatório.
- R-2 refund duplo/período errado → gate por transição + decrementUsageForPeriod.
- R-3 race sweeper×worker → status-guards (`WHERE status='PENDING'`) + lock de linha PG + re-enqueue idempotente.
- R-4 apagar input/output necessário → só terminais/expirados; M-03.
- R-5 cron não roda → kill-switch dedicado + gauge stale.
