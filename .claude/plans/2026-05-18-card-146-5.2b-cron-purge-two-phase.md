#### AMB-1 — Como cron deleta no Storage

**Contexto:** `file_history.storagePath` é `{userId}/{yyyy-mm-dd UTC}/{jobId}.{ext}` (string string brand `UserScopedPath`). O adapter foi desenhado **user-scoped por construção** (`deleteForUser({userId, jobId, ext})`) pra prevenir IDOR — caller nunca passa path raw.

Cron tem só `storagePath` do DB. Três opções:

- **A — Parsear o `storagePath` no cron** → split `/`, extract `userId/date/jobId.ext`, chamar `deleteForUser({userId, jobId, ext})`. Caller revalida shape com asserts existentes.
  - **Prós:** zero mudança no adapter. Reusa todas as validações key-builder.
  - **Contras:** parsing fragil; muda no futuro (ex: subfolder) = bug silencioso. Acopla cron à estrutura interna do path.
  - **Custo:** ~10 LoC + 2 testes parse.

- **B — Adicionar `removeByPath(path: UserScopedPath): Promise<{ deleted: boolean }>`** ao `StorageAdapter`.
  - **Prós:** API limpa, alinhada com casos "delete by stored reference". `UserScopedPath` brand garante caller passou path validado em algum momento.
  - **Contras:** quebra hard-rule "adapter user-scoped por construção" — caller (cron) recebe path do DB e o brand não prova nada (DB pode estar corrompido). **Mitigação:** método valida internamente via `assertNoPathTraversal` + regex `{uuidv4}/{yyyy-mm-dd}/{cuid}.{ext}` antes de chamar Supabase. Custo +5 LoC vs A.
  - **Custo:** ~25 LoC adapter + 1 LoC interface + 4 testes (path válido, traversal rejeitado, 404 → success, erro Supabase).

- **C — Aceitar path raw no `deleteForUser` como overload** (`{path}` mutually-exclusive com `{userId, jobId, ext}`).
  - **Prós:** zero novo método.
  - **Contras:** overload acopla shape do args — TS discriminated union pesado, perde benefício do user-scoped por construção em **toda** chamada existente. Refactor invasivo só pra cron.

**Recomendação: B**. `removeByPath` documentado como "uso restrito a callers que recebem path do DB (cron de purga)". Adapter valida o shape **antes** de chamar Supabase. Mantém hard-rule pros caminhos quentes (upload/signed URL via user-scoped builder) e abre uma porta controlada pro cron. Confidence: high.

**Severity se errar:** ALTO (impacto direto em segurança do adapter — A é o caminho fácil que vira CVE no futuro com schema novo).

#### AMB-2 — Schedule UTC vs BRT

**Contexto:** descrição do card diz "03:00 BRT". Scheduler é UTC (`timezone: 'UTC'` em `cron.schedule`). BRT é UTC-3 (sem DST desde 2019).

- **A — `'0 3 * * *'` UTC = 00:00 BRT** (3h da manhã UTC → 0h BRT).
  - Prós: literal do card.
  - Contras: 00:00 BRT é hora de pico (jobs noturnos legítimos: backup, billing, relatórios). Worst time pra purge competir por DB.

- **B — `'0 6 * * *'` UTC = 03:00 BRT** ✅
  - Prós: 03:00 BRT é vale de tráfego (poucos users acordados); minimiza contention. Alinhado com convenção operacional "jobs pesados rodam 3am local".
  - Contras: requer comentário explícito no código (operador futuro pode achar que "6 UTC" é typo).

- **C — Schedule via env (`PURGE_CRON_SCHEDULE`)**.
  - Prós: tunável sem deploy.
  - Contras: superfície a mais, fail-fast no boot exige Zod regex de cron expression. Over-engineering pra valor fixo. Discovery card.

**Recomendação: B** com comentário fixo `'0 6 * * *' // 03:00 BRT (UTC-3)`. Confidence: high.

**Severity se errar:** MÉDIO (operacional — má janela degrada UX mas não viola LGPD).

#### AMB-4 — Quem tira do dead-letter

**Contexto:** após 5 tentativas falhas, row fica com `deleted_at != null AND purge_attempts >= 5`. Cron pode ignorar (filtro `purge_attempts < 5`) ou retomar.

- **A — Cron retoma automaticamente** após N horas (ex: 24h).
  - Prós: self-healing.
  - Contras: se a falha era determinística (corrupção, bug), retry infinito polui Sentry + audit_log_legal.

- **B — Admin manual** via SQL ou endpoint dedicado (`POST /admin/jobs/history-purge/retry-deadletter`).
  - Prós: humano avalia root cause antes de retry (runbook obriga investigar Sentry/logs).
  - Contras: requer intervenção (mas o cenário é raro — alerta Sentry critical).
  - **Mitigação imediata:** runbook `dead-letter-purge.md` (novo) documenta query SQL `UPDATE file_history SET purge_attempts = 0 WHERE id IN (...)` com pré-check de causa.

- **C — Dead-letter table** (`file_history_dead_letter`) com cron próprio de reprocess weekly.
  - Prós: separa fluxo feliz do degradado.
  - Contras: schema novo + migration + lifecycle complexo. Over-engineering pra cenário esperado <1×/mês.

**Recomendação: B**. Discovery card `purge-deadletter-admin-endpoint` (BAIXA, P) caso o cenário se torne frequente; senão SQL manual via runbook basta. Confidence: high.

**Severity se errar:** MÉDIO (operacional — A polui Sentry; C é complexidade prematura).

## 6. Decisões & trade-offs

### D-A — Localização do handler: `src/jobs/`
Diretório novo top-level pra background workers (cron handlers, queue workers futuros). Não em `src/scheduler/` (infra) nem `src/modules/history/` (módulo HTTP). Confidence: high. Reversível: sim (mover é refactor simples).

### D-B — `removeByPath` no adapter (vs parse no cron) — AMB-1
Recomendado B (ver §1.3 AMB-1). Adapter mantém isolamento contractual; novo método com validação interna explícita. Confidence: high. Reversível: parcialmente (remover método quebra cron — exige plano).

### D-C — Schedule `'0 6 * * *'` UTC (= 03:00 BRT) — AMB-2
Hard-coded com comentário. Confidence: high. Reversível: trivial (literal no `registerCronJob`).

### D-D — Estado de cron runs in-memory + gauge — AMB-3
F5 do #145 + gauge novo. `audit_log_legal` é fallback forense persistente. Discovery card pra tabela `cron_runs` futura. Confidence: high. Reversível: sim (adicionar tabela depois é não-breaking).

### D-E — Dead-letter via admin manual + runbook — AMB-4
Sem cron auto-retry; humano avalia. Discovery card pra endpoint dedicado. Confidence: high.

### D-F — Bootstrap em módulo dedicado (`src/scheduler/jobs.bootstrap.ts`)
Não em `server.ts` direto — facilita testes (server.ts já é hard-to-test pelo `app.listen`). Função `bootstrapCronJobs()` exportada chamada antes do `app.listen`. Confidence: high.

### D-G — Batch sleep 100ms entre commits
Reduz pressure no DB; ~3 batches/seg max = 1500 rows/seg (vs 500k rows DELETE atomic que travaria index). Confidence: high.

### D-H — Statement timeout 30s por sessão (via `$executeRawUnsafe('SET LOCAL ...')` como primeira query da tx)
Cap defensivo — batch normal demora <2s; 30s indica row corrompida/lock externo. Confidence: high.

### D-I — Sentry alert level `purge.dead_letter` = ERROR (não warning)
Dead-letter é incidente real, não ruído. Alerta deve acordar quem está de plantão. Confidence: high.
