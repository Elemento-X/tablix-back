## 3. Hard requirements por agente

### 3.1 @dba
- Migration **expand-only** — sem `DROP` ou `RENAME`
- UNIQUE constraint `(user_id, threshold, period)` via index dedicado (não inline) — facilita drop futuro
- CHECK constraint: `threshold IN (70, 90)` (enum like; pré-go-live podemos adicionar valores via migration)
- CHECK constraint: `period ~ '^\d{4}-(0[1-9]|1[0-2])$'` (formato YYYY-MM válido)
- FK: `user_id REFERENCES users(id) ON DELETE CASCADE` (LGPD — purge user purga histórico de alertas)
- Índice composto cobre dedupe lookup `(user_id, threshold, period)` (já é a UNIQUE — ok)
- Coluna `sent_at TIMESTAMPTZ NOT NULL DEFAULT now()` — timezone-aware
- RLS habilitada: `ENABLE ROW LEVEL SECURITY` + policy única `service_role only` (cron e admin operam via service-role; nunca user-context)
- Comentários SQL (`COMMENT ON TABLE/COLUMN`) com link pra #147
- Verificar EXPLAIN ANALYZE da query SELECT PRO ativos não faz seq scan em users
- Reusar `Token` model existente — filtro PRO ativo = `Token.expiresAt > now AND Token.revokedAt IS NULL`

### 3.2 @security
- LGPD: alerta envia email do user. Templates **não** contêm dados de outros users. `to: user.email` direto, sem CC/BCC.
- Logs: `userId` (UUID, OK porque já está em logs de auth/billing), `usagePercent` (number 0-100), `threshold` (70|90), `period` (YYYY-MM). **NÃO LOGAR**: `user.email`, `unificationsCount` absoluto, qualquer campo de `User` ou `Token`.
- Métricas agregadas only: `users_above_threshold_total{threshold,job}` (counter) e gauge `users_above_threshold{threshold}` (atual). NÃO emitir métricas por user.
- Dry-run em prod: warning Sentry (já cobre via `cron.purge.dry_run.start` pattern → adicionar `cron.quota_alert.dry_run.start` no `ALERTABLE_IN_PROD_ONLY`)
- Idempotência: 3 instâncias do worker simultâneas → lock distribuído garante 1 (reusa). Crash mid-batch → próximo run reconcilia (alertas pendentes têm `INSERT...ON CONFLICT DO NOTHING` natural). Não há split-brain risk porque INSERT é atomic + UNIQUE absorve duplicação.
- Anti-spam: dedupe mensal via UNIQUE constraint. Sem isso, restart do worker no meio do batch reenvia email.
- Resend API key não-logada (já protegido em REDACT_PATHS de #145 carryover)
- Template HTML: sem inline JS (já é padrão dos templates atuais), sem `<img src="http://...">` (apenas inline CSS + texto), sem URLs externas exceto `env.FRONTEND_URL`
- Email contém `unsubscribe` link? NÃO — alertas de quota PRO são essenciais ao serviço (transacional, não marketing). LGPD/CASL distingue transacional de marketing. Documentar no template.

## 5. Migration SQL completa

```sql
-- supabase/migrations/<ts>_add_quota_alerts_sent.sql
-- Card #147 (5.2c) F1 — tabela de dedupe de alertas de quota mensal.
--
-- Pattern: UNIQUE(user_id, threshold, period) + INSERT...ON CONFLICT DO NOTHING
-- garante atomicamente "1 alerta por threshold por mês por user" mesmo sob
-- concorrência (crash mid-batch, 2 workers, etc).
--
-- @owner: @dba + @security
-- @card: #147 (5.2c) F1

CREATE TABLE quota_alerts_sent (
  id          UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID         NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  threshold   INTEGER      NOT NULL,
  period      VARCHAR(7)   NOT NULL,   -- YYYY-MM UTC
  sent_at     TIMESTAMPTZ  NOT NULL DEFAULT now(),

  CONSTRAINT quota_alerts_threshold_check
    CHECK (threshold IN (70, 90)),

  CONSTRAINT quota_alerts_period_format_check
    CHECK (period ~ '^\d{4}-(0[1-9]|1[0-2])$'),

  CONSTRAINT quota_alerts_unique_user_threshold_period
    UNIQUE (user_id, threshold, period)
);

COMMENT ON TABLE quota_alerts_sent IS
  'Dedupe de alertas de quota mensal (Card #147, 5.2c). 1 row = 1 alerta enviado. UNIQUE(user_id, threshold, period) garante anti-duplicação por user/threshold/mês.';

COMMENT ON COLUMN quota_alerts_sent.threshold IS
  '70 ou 90 (% de uso da quota PRO mensal). CHECK constraint enforça enum-like.';

COMMENT ON COLUMN quota_alerts_sent.period IS
  'YYYY-MM em UTC, alinhado com usage.service.getCurrentPeriod(). Reset mensal natural via period nova → ON CONFLICT DO NOTHING permite reenvio.';

COMMENT ON COLUMN quota_alerts_sent.sent_at IS
  'Timestamp do envio (audit). NOT logged as PII porque user_id já está em logs.';
```

```sql
-- supabase/migrations/<ts>_quota_alerts_sent_rls.sql
-- Card #147 (5.2c) F1 — RLS service-role only.
--
-- Cron + admin operam via service-role (bypass RLS implícito).
-- Nenhum user-context tem acesso — alertas são metadata interna, não dado do usuário.
-- (User pode ver suas próprias quotas via GET /usage do Card 4.1.)

ALTER TABLE quota_alerts_sent ENABLE ROW LEVEL SECURITY;

CREATE POLICY quota_alerts_service_role_only ON quota_alerts_sent
  FOR ALL
  TO public
  USING (false)
  WITH CHECK (false);

COMMENT ON POLICY quota_alerts_service_role_only ON quota_alerts_sent IS
  'Deny-all pra public/authenticated. Service-role (Prisma direct connection) bypass RLS implícito — não precisa de policy explícita pra ALLOW.';
```

  @@unique([userId, threshold, period], map: "quota_alerts_unique_user_threshold_period")
  @@map("quota_alerts_sent")

## 12. Pre-mortem (3 falhas mais prováveis em 3 meses)

### 12.1 Email duplicado (mais provável)
**Cenário:** restart do worker no meio do batch → re-roda → user recebe 2 emails idênticos.
**Por que provável:** cron handlers podem falhar Mid-batch (DB hiccup, Redis lag).
**Mitigação:** UNIQUE constraint + `ON CONFLICT DO NOTHING` é hard guarantee — testado em F3 cenário 2 + property-based. **Trade-off A-8**: email Resend falhar → INSERT mesmo assim (1 perdido > 30 duplicados).
**Detecção:** alerta Sentry `cron.quota_alert.email_failed` permite identificar falsos perdidos.

### 12.2 Spam de quota a 70% para users que oscilam 65→75→65 mensalmente
**Cenário:** user faz 18 unificações em 1 dia (cross 70%), depois fica 29 dias sem usar. Recebe alerta em todo mês onde brevemente cruzou.
**Por que provável:** uso real de SaaS é spiky.
**Mitigação:** É **comportamento esperado** — 1 alerta/mês não é spam (escopo do dedupe mensal). User-controlled opt-out fora de escopo (item declarado).
**Detecção:** se taxa de complaint chegar, criar card `quota-alert-frequency-tuning` para hyster-banding.

### 12.3 Cron parado silenciosamente em prod, descoberto quando user atinge 100% sem aviso prévio
**Cenário:** dry-run esquecido em prod, ou kill-switch desligado por engano em outro card, ou lock orphan.
**Por que provável:** ops humano. Card #146 já passou por isso (dry-run em prod warning).
**Mitigação:**
- Sentry alert para `dry_run.start` em prod (warning level, sinal claro)
- Stale gauge: `users_above_threshold.lastUpdatedAt > 25h` → runbook `quota-alert-stuck.md` orienta
- Health check `/admin/jobs/list` expõe `lastDurationMs` e `runs_total` por job
- Discovery card `cron-stuck-auto-alert` (Backlog, P) — alerta se cron sem run > 25h pode ser feature de plataforma
